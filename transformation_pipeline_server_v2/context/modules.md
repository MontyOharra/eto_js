# Module Definitions & Catalog Population — Design Overview

> Audience: Claude (to read and understand; no code changes requested).
> Scope: How modules are defined in Python, how dynamic I/O works, how the UI is driven by JSON Schema, and how a dev “sync” populates a module catalog in the database.

---

## 1) What a “Module” Is

A **module** is a reusable processing block with:

* **Kind**: `transform`, `action`, or `logic` (different execution semantics).
* **I/O shape**: one **list of inputs** and one **list of outputs**. Each side can be:

  * fixed (e.g., exactly 1 input, 1 output), or
  * dynamic (variable count within min/max constraints).
* **Config**: a Pydantic model that defines the module’s user-configurable options (and yields a JSON Schema for the UI).
* **Run function**: pure function for transforms/logic; side-effecting for actions.

Pins (I/O “nodes”) are concrete at *pipeline* level and identified by **random IDs**. Ordering within each side is expressed via `position_index` (0..N−1).

---

## 2) Dynamic I/O Model (single list per side)

Each module declares **I/O side rules**:

```python
# types.py (conceptual)
from typing import Optional, Literal, Union, Dict
from pydantic import BaseModel

Scalar = Literal["str","int","float","datetime","bool"]
VarType = Dict[str, object]  # {"mode":"variable","allowed":[...]}

class DynamicSide(BaseModel):
    allow: bool = True
    min_count: int = 0
    max_count: Optional[int] = None   # None = unbounded
    type: Union[Scalar, VarType] = "str"

class ModuleMeta(BaseModel):
    inputs: DynamicSide
    outputs: DynamicSide
```

* **Validator** enforces side rules against the *pipeline JSON*:

  * number of pins within `min..max`,
  * all pins have correct `direction`,
  * `type` matches the side’s rule,
  * `position_index` is 0..N−1, unique, and contiguous.
* **Executor** doesn’t need the rules; it receives concrete pins already.

No grouping is needed because there’s **only one dynamic list per side**.

---

## 3) JSON Schema Drives the UI (no separate “ui hints” table)

The UI renders module config forms **purely from the module’s JSON Schema** generated by Pydantic, with a tiny `x-ui` extension for display-only hints:

* **Widget mapping** (simple rules)

  * `type: "boolean"` → switch/checkbox
  * `enum: [...]` → dropdown
  * `type: "string"` → text input

    * with `x-ui.widget: "textarea"` → multi-line text area
* **Field order** (optional)

  * root `x-ui.order: ["fieldA", "fieldB", ...]`

### Example: schema extras in Pydantic v2

```python
from pydantic import BaseModel, Field, ConfigDict

class LlmConfig(BaseModel):
    model: str = Field(
        ...,
        description="LLM to use",
        json_schema_extra={"enum": ["claude-3.5", "gpt-4o", "gpt-4.1-mini"]}
    )
    temperature: float = Field(0.1, ge=0, le=1, description="0..1")
    prompt: str = Field(
        ...,
        description="Use {input1},{input2},… by input order.",
        json_schema_extra={"x-ui": {"widget": "textarea"}}
    )

    model_config = ConfigDict(
        json_schema_extra={"x-ui": {"order": ["model","temperature","prompt"]}}
    )
```

The **catalog** serves this JSON Schema directly. The frontend reads it and renders appropriate controls.

## 4) Base Classes & Example Modules

### Base classes

```python
# contracts.py (essentials)
from typing import Any, Dict, Literal
from pydantic import BaseModel
from types import ModuleMeta  # from §2

class CommonCore:
    id: str
    version: str
    title: str
    description: str
    kind: Literal["transform","action","logic"] = "transform"

    class ConfigModel(BaseModel): ...
    @classmethod
    def meta(cls) -> ModuleMeta: ...
    @classmethod
    def config_schema(cls) -> dict:
        return cls.ConfigModel.model_json_schema()

    # Optional per-module wiring checks; return a list of errors (or [])
    @classmethod
    def validate_wiring(cls, module_instance_id: str, config: Dict[str, Any],
                        instance_inputs: list[dict], instance_outputs: list[dict],
                        upstream_of_input: Dict[str,str]) -> list[dict]:
        return []

    # Execution: return { out_node_id -> value }
    def run(self, inputs: Dict[str, Any], cfg: BaseModel, context: Any|None=None) -> Dict[str, Any]:
        raise NotImplementedError

class TransformModule(CommonCore): kind = "transform"
class ActionModule(CommonCore):    kind = "action"
class LogicModule(CommonCore):     kind = "logic"
```

### Example: `TextClean` (transform, 1 in → 1 out)

```python
# modules/builtins/text_clean.py
from typing import Any, Dict
from pydantic import BaseModel, Field
from contracts import TransformModule, ModuleMeta, DynamicSide

class TextCleanConfig(BaseModel):
    strip: bool = Field(True, description="Trim leading/trailing whitespace.")
    collapse_space: bool = Field(True, description="Collapse whitespace to single spaces.")
    lowercase: bool = Field(False, description="Convert to lowercase.")

class TextClean(TransformModule):
    id="text_clean"; version="1.0.0"
    title="Text Clean"; description="Trim/normalize whitespace."

    @classmethod
    def meta(cls) -> ModuleMeta:
        return ModuleMeta(
            inputs=DynamicSide(allow=True, min_count=1, max_count=1, type="str"),
            outputs=DynamicSide(allow=True, min_count=1, max_count=1, type="str"),
        )

    def run(self, inputs: Dict[str, Any], cfg: TextCleanConfig, context=None) -> Dict[str, Any]:
        # exactly one input and one output
        (in_id, value), = inputs.items()
        s = "" if value is None else str(value)
        if cfg.strip: s = s.strip()
        if cfg.collapse_space:
            import re; s = re.sub(r"\s+", " ", s)
        if cfg.lowercase: s = s.lower()
        out_id = context.instance_ordered_outputs[0]["node_id"]
        return { out_id: s }
```

### Example: `LlmParser` (transform, N inputs → 1 output)

```python
# modules/builtins/llm_parser.py
from typing import Any, Dict
from pydantic import BaseModel, Field, ConfigDict
from contracts import TransformModule, ModuleMeta, DynamicSide

class LlmConfig(BaseModel):
    model: str = Field(..., json_schema_extra={"enum": ["claude-3.5","gpt-4o","gpt-4.1-mini"]})
    temperature: float = Field(0.1, ge=0, le=1)
    prompt: str = Field(..., json_schema_extra={"x-ui": {"widget": "textarea"}})
    model_config = ConfigDict(json_schema_extra={"x-ui": {"order": ["model","temperature","prompt"]}})

class LlmParser(TransformModule):
    id="llm_parser"; version="2.2.0"
    title="LLM Parser"; description="Combine multiple text inputs using a prompt."

    @classmethod
    def meta(cls) -> ModuleMeta:
        return ModuleMeta(
            inputs=DynamicSide(allow=True, min_count=1, max_count=None, type="str"),
            outputs=DynamicSide(allow=True, min_count=1, max_count=1, type="str"),
        )

    def run(self, inputs: Dict[str, Any], cfg: LlmConfig, context=None) -> Dict[str, Any]:
        # executor provides ordered pin metadata for convenience
        ordered_ins  = context.instance_ordered_inputs   # [(node_id, value), ...] by position_index
        ordered_outs = context.instance_ordered_outputs  # [{"node_id": ...}, ...]
        prompt = cfg.prompt
        for i, (_, val) in enumerate(ordered_ins, start=1):
            prompt = prompt.replace(f"{{input{i}}}", str(val))
        result = "...model output..."
        return { ordered_outs[0]["node_id"]: result }
```

## 5) In-Memory Registry & Dev “Sync” (DB population)

### Registry

```python
# registry.py
from typing import Dict, Type
from contracts import CommonCore

_REGISTRY: Dict[str, Type[CommonCore]] = {}

def register_module(cls: Type[CommonCore]) -> Type[CommonCore]:
    ref = f"{cls.id}:{cls.version}"
    if ref in _REGISTRY:
        raise ValueError(f"Duplicate module ref: {ref}")
    _REGISTRY[ref] = cls
    return cls

def get(ref: str) -> Type[CommonCore]:
    return _REGISTRY[ref]

def all_modules() -> dict[str, Type[CommonCore]]:
    return dict(_REGISTRY)
```

Use it as a decorator:

```python
# modules/builtins/text_clean.py (continued)
from registry import register_module
@register_module
class TextClean(TransformModule):
    ...
```

### Catalog row shape (what the UI/validator needs)

For each registered class:

* `module_ref` (`id:version`), `module_kind`, `name`, `description`, (optional) `color`, `category`
* `meta` (I/O side rules; see §2)
* `config_schema` (Pydantic JSON Schema with any `x-ui` extras)
* `handler_name` (`"python.module.path:ClassName"`)
* `is_active`

### Dev “Sync” script (example)

```python
# tools/sync_modules.py
import json
from typing import Type
from registry import all_modules
from contracts import CommonCore

def db_add_to_modules(payload: dict) -> None:
    # Replace with a real upsert into module_catalog
    print(json.dumps(payload, indent=2))

def class_to_catalog_row(cls: Type[CommonCore]) -> dict:
    meta_json = cls.meta().model_dump(mode="json")
    return {
        "id": cls.id,
        "version": cls.version,
        "module_ref": f"{cls.id}:{cls.version}",
        "name": cls.title,
        "description": cls.description,
        "module_kind": cls.kind,  # "transform" | "action" | "logic"
        "color": getattr(cls, "color", "#3B82F6"),
        "category": getattr(cls, "category", "Processing"),
        "meta": meta_json,
        "config_schema": cls.config_schema(),
        "handler_name": f"{cls.__module__}:{cls.__name__}",
        "is_active": True
    }

def main():
    # Ensure builtins are imported so @register_module runs
    import modules.builtins.text_clean   # noqa
    import modules.builtins.llm_parser   # noqa
    # ... import other modules here ...

    for ref, cls in all_modules().items():
        db_add_to_modules(class_to_catalog_row(cls))

if __name__ == "__main__":
    main()
```

The “DB population” step is an **upsert** per module class into a `module_catalog` table (or equivalent). The important part is that the **catalog row** is derived exclusively from the **code** (module meta + config schema + handler path).

---

## 6) What the Frontend Consumes

The builder calls something like `GET /modules` and receives a list of catalog rows. From each row it uses:

* `meta.inputs` / `meta.outputs` to decide whether to show “Add input/output” and to validate local counts and types.
* `config_schema` to render the config form:

  * `type`, `enum`, `description`, `default` (standard JSON Schema),
  * `x-ui.widget: "textarea"` (optional extension),
  * `x-ui.order` (optional extension).
* `module_kind`, `name`, `description`, `color`, `category` for palette visuals.

No separate “UI hints” column is needed.

---

## 7) Summary

* **Modules** are Python classes with:

  * **kind**, **I/O side rules** (`ModuleMeta`), **Pydantic config**, and a **run** method.
* **Pins** at pipeline level use **random IDs**; **order** is via **`position_index`** per side.
* **UI** is driven entirely by the **Pydantic JSON Schema**, with a tiny `x-ui` extension for textarea and (optionally) field order.
* A **registry** collects module classes; a **dev sync** script serializes each into a **catalog row** and **populates the database**.
* The catalog provides the **single source of truth** for the builder and validator: I/O constraints, config schema, and the handler to invoke at runtime.
